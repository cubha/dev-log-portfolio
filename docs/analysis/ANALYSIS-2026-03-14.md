## 2. 성능 병목 핵심 진단

### 2-1. 가장 느린 원인: 캐싱 전략 완전 부재

**프로젝트 전체에 캐싱 메커니즘이 단 하나도 적용되지 않았다.**

- `unstable_cache` / `use cache` 사용: **0건**
- `React.cache()` 래핑: **0건**
- `revalidate` / `revalidateTag` 사용: **0건** (Server Action의 `revalidatePath` 제외)
- `force-dynamic` 선언: **3개 페이지** (Contact, Projects, Admin)

| 페이지 | 캐싱 설정 | DB 쿼리 수 | 체감 영향 |
|---|---|---|---|
| `/` (홈) | Next.js 기본 | 최대 2 | 낮음 |
| `/about` | 없음 | **7** (병렬) | **중간** |
| `/contact` | `force-dynamic` | **최대 9** (순차 포함) | **매우 높음** |
| `/projects` | `force-dynamic` | 3 | **높음** |
| `/projects/[slug]` | SSG + 런타임 컴파일 | 0 (shiki CPU) | **높음** |
| `/admin/dashboard` | `force-dynamic` | **9** (병렬) | 중간 (관리자 전용) |

**매 요청마다 Supabase로 네트워크 왕복이 발생하므로, 사용자가 페이지를 이동할 때마다 "빈 화면 → 로딩 → 데이터 표시" 지연이 반복된다.**

### 2-2. `auth.getUser()` 중복 호출

`auth.getUser()`는 Supabase Auth 서버로 매번 네트워크 요청을 보내는 비싼 함수다. 동일 요청 범위(같은 페이지 렌더링)에서 `React.cache()`로 감싸지 않아 중복 호출이 발생한다.

| 페이지 | `auth.getUser()` 호출 횟수 | 설명 |
|---|---|---|
| `/contact` | **3회** | `getCurrentUserRole()` + `page.tsx` 직접 + `GuestbookList` 내부 |
| `/projects` | 1회 | `getCurrentUserRole()` |
| Server Actions | **매 호출마다 1회** | 7개 Server Action 각각 독립 호출 |

### 2-3. Contact 페이지 — 가장 심각한 병목

Contact 페이지는 프로젝트에서 가장 느린 페이지다:

```
요청 시작
  ├─ getCurrentUserRole()
  │   ├─ auth.getUser()          [네트워크 #1]
  │   └─ profiles.select('role') [네트워크 #2]
  ├─ auth.getUser()              [네트워크 #3] ← 중복!
  ├─ profiles.select('user_id')  [네트워크 #4]
  ├─ contact_links.select('*')   [네트워크 #5]
  └─ GuestbookList (서버 컴포넌트)
      ├─ guestbook.select + count [네트워크 #6]
      ├─ auth.getUser()           [네트워크 #7] ← 또 중복!
      ├─ guestbook_comments       [네트워크 #8] ← 순차 실행
      └─ guestbook_likes          [네트워크 #9] ← 순차 실행
```

**9회의 Supabase 네트워크 왕복**이 순차적으로 실행되며, 캐싱이 없어 매 방문마다 반복된다.

## 3. 개별 이슈 상세 분석

### 3-1. Technical Skills 펼침 시 느린 원인

**원인: 클라이언트 사이드 대량 DOM 렌더링 + Framer Motion stagger 애니메이션**

`SkillsSection.tsx`는 클라이언트 컴포넌트로:
- 펼침 시 모든 skills를 **한 번에 DOM에 마운트**
- 각 스킬 카드가 개별 `motion.div`로 래핑
- `stagger delay = idx * 0.06` → 100개 카드면 6초에 걸친 순차 애니메이션
- `whileInView` + `viewport` 감시 → 각 카드마다 IntersectionObserver 등록
- 추가 API 호출은 없음 (데이터는 이미 About 페이지에서 fetch 완료)

**핵심**: 네트워크가 아니라 **렌더링/애니메이션 성능**이 병목.

### 3-2. 방명록 댓글 불러오기 느린 원인

**원인 1: Server Action `getGuestbookComments()` 내부 3회 순차 쿼리**
```
auth.getUser()                    [네트워크 #1]
guestbook_comments.select(...)    [네트워크 #2]
guestbook_comment_likes.select(...)  [네트워크 #3]
```

**원인 2: 댓글 페이지네이션 없음**
- 한 방명록 항목의 모든 댓글을 한번에 로드
- 댓글이 많아질수록 선형적으로 느려짐

**원인 3: `fetchComments()` + `router.refresh()` 이중 갱신**
- 댓글 등록/삭제 시 Server Action으로 클라이언트 상태 갱신 + `router.refresh()`로 서버 컴포넌트 전체 재렌더링
- 두 갱신이 동시에 발생하여 불필요한 이중 작업

### 3-3. 페이지 네비게이션이 전반적으로 느린 원인

1. **캐싱 없음**: 모든 페이지가 매번 Supabase fresh fetch
2. **`force-dynamic`**: Contact/Projects 페이지는 SSG/ISR 완전 비활성화
3. **`select('*')` 남용**: 필요없는 컬럼까지 전송 (특히 `about_profiles`의 큰 JSON 컬럼)
4. **순차 쿼리**: 병렬화 가능한 쿼리가 순차 실행
5. **`loading.tsx`는 존재**: 4개 라우트에 로딩 UI 있으나, 근본적인 데이터 페칭 지연을 해결하지 못함

## 4. 아키텍처 평가

### 구조 개요
```
서버 컴포넌트 (page.tsx)
  ├─ getCurrentUserRole()     ← 모든 페이지 공통
  ├─ Supabase 쿼리 N개        ← 캐싱 없음
  └─ 클라이언트 컴포넌트 렌더
      ├─ Framer Motion 애니메이션
      ├─ Jotai 상태 (최소 사용)
      └─ 추가 Server Action 호출 (온디맨드)
```

### 평가
| 항목 | 평가 | 근거 |
|---|---|---|
| 레이어 분리 | **보통** | 서버/클라이언트 분리는 양호하나 데이터 레이어 캐싱 전략 부재 |
| 패턴 일관성 | **보통** | `select('*')` 일관적이나 비효율적, auth 중복 호출 패턴 반복 |
| 확장성 | **낮음** | 캐싱 없이 데이터 증가 시 선형 성능 저하 |
| 테스트 가능성 | **낮음** | 테스트 파일 0개 |

### 강점
- `simple-icons` named import로 tree-shaking 최적화 양호
- `generateStaticParams()` MDX 정적 생성 구현
- `loading.tsx` 4개 라우트에 존재
- `Promise.all` 병렬 쿼리 일부 적용 (About, Admin Dashboard)
- 서버/클라이언트 컴포넌트 분리 원칙 준수
- `next/image` 전반적 사용

## 6. 개선 로드맵

### 즉시 개선 (Quick Win) — 코드 변경 최소, 체감 효과 큼

- [ ] **`auth.getUser()` 중복 제거**: `React.cache()`로 감싸서 동일 요청 범위 내 1회만 호출
  ```ts
  // src/utils/supabase/server.ts 또는 별도 파일
  import { cache } from 'react'
  export const getCachedUser = cache(async () => {
    const supabase = await createClient()
    return supabase.auth.getUser()
  })
  ```
  예상 효과: Contact 페이지 네트워크 왕복 9→7회 (즉시 ~20% 감소)

- [ ] **GuestbookList 쿼리 병렬화**: 댓글 수/좋아요 쿼리를 `Promise.all()`로 병렬 실행
  ```ts
  const [{ data: commentRows }, { data: likeRows }] = await Promise.all([
    supabase.from('guestbook_comments').select('guestbook_id').in('guestbook_id', entryIds),
    supabase.from('guestbook_likes').select('guestbook_id').in('guestbook_id', entryIds),
  ])
  ```
  예상 효과: Contact 페이지 순차 대기 시간 ~30% 감소

- [ ] **`select('*')` → 필요 컬럼만 명시**: 특히 `about_profiles`(큰 JSON), `projects`(content 컬럼 불필요)
  예상 효과: 네트워크 payload 감소

- [ ] **Skills 애니메이션 최적화**: stagger 제거 또는 `staggerChildren` 값 축소 (0.06→0.02), `whileInView` 제거
  예상 효과: Skills 펼침 즉시 반응

### 단기 개선 (1~2주) — 구조적 캐싱 전략 도입

- [ ] **`/projects` 페이지 ISR 전환**: `force-dynamic` 제거 → `export const revalidate = 60` (1분 캐싱)
  - 프로젝트 추가는 자주 발생하지 않으므로 ISR이 적합
  - Server Action에서 `revalidatePath('/projects')` 호출로 즉시 무효화

- [ ] **`/about` 페이지 ISR 전환**: `export const revalidate = 300` (5분 캐싱)
  - Skills/Experience/Education 데이터는 거의 변경되지 않음
  - 관리자 수정 시 `revalidatePath('/about')` 호출

- [ ] **`getCurrentUserRole()` 결과를 `React.cache()`로 캐싱**
  ```ts
  export const getCurrentUserRole = cache(async (): Promise<'admin'|'user'|'anon'> => {
    // 기존 로직
  })
  ```

- [ ] **shiki 하이라이터 싱글턴**: `createHighlighter()`로 1회 생성 후 재사용
  ```ts
  import { createHighlighter } from 'shiki'
  let highlighter: Awaited<ReturnType<typeof createHighlighter>> | null = null
  export async function getHighlighter() {
    if (!highlighter) {
      highlighter = await createHighlighter({ themes: ['github-dark'], langs: [...] })
    }
    return highlighter
  }
  ```

- [ ] **`getGuestbookComments()` 내부 쿼리 병렬화**: `auth.getUser()` 후 댓글/좋아요 조회를 `Promise.all()`로

### 중장기 개선 (1개월+) — 아키텍처 개선

- [ ] **Contact 페이지 Streaming/Suspense 분리**: 방명록 리스트를 별도 `<Suspense>` 경계로 감싸서 contact_links는 즉시 렌더, 방명록은 스트리밍
  ```tsx
  <Suspense fallback={<GuestbookSkeleton />}>
    <GuestbookList ... />
  </Suspense>
  ```

- [ ] **`page_views` DB 사이드 집계**: Supabase RPC (PostgreSQL 함수)로 GROUP BY 집계 이동
  ```sql
  CREATE OR REPLACE FUNCTION get_daily_page_views(start_date timestamptz)
  RETURNS TABLE(day date, count bigint) AS $$
    SELECT date_trunc('day', created_at)::date, COUNT(*)
    FROM page_views WHERE created_at >= start_date
    GROUP BY 1 ORDER BY 1
  $$ LANGUAGE sql STABLE;
  ```

- [ ] **Spotify access_token 캐싱**: Route Handler 내 모듈 수준 변수에 토큰 + 만료시각 저장, 만료 전까지 재사용
